# Landing Page Wireframe

Det här dokumentet kompletterar:

- `docs/marketing/landing-page-strategy.md`
- Issue #222
- Epic #223

Syftet är att ge Codex en konkret struktur att bygga efter, så att landningssidan inte bara optimeras textmässigt utan även får rätt layout, prioritering och mobilupplevelse.

---

## Övergripande princip

Landningssidan ska byggas för mobil först.

Desktop får gärna vara mer visuell och luftig, men mobilversionen är viktigast eftersom många småföretagare kommer öppna sidan från mail, sociala medier eller mobilen.

Hero-sektionen ska inte bära hela SEO-arbetet. Den ska få besökaren att förstå värdet och vilja fortsätta.

SEO-bredden ska ligga längre ned på sidan i strukturerade sektioner.

---

## Sidstruktur

```text
1. Top navigation
2. Hero
3. Problemigenkänning
4. Tre huvudproblem och lösningar
5. Så fungerar det
6. Funktioner kopplade till problem
7. För vilka verksamheter?
8. Kontrollant / inspektörsvy
9. Vanliga frågor
10. Kunskapsbank / fördjupning
11. Slut-CTA
```

## 1. Top navigation

### Desktop

- Logotyp / Min Egenkontroll
- Eventuella ankarlänkar:
  - Problem
  - Så fungerar det
  - Vanliga frågor
- Primär knapp: `Kom igång`
- Sekundär knapp: `Logga in`

### Mobil

- Logotyp / Min Egenkontroll
- `Logga in`
- Eventuellt menyikon om nödvändigt
- Undvik att navigationen tar för mycket höjd

---

## 2. Hero

### Mål

Besökaren ska inom några sekunder förstå:

1. Vad produkten gör.
2. Vilket problem den löser.
3. Vad nästa steg är.

### Rekommenderad copy-riktning

#### H1

```text
Digital egenkontroll som personalen faktiskt hinner med
```

Alternativ:

```text
Egenkontroll direkt i mobilen – utan pärmar och papperslistor
```

#### Ingress

```text
Min Egenkontroll visar vad som ska göras, låter personalen dokumentera på plats och samlar historiken så den är redo när kontrollanten kommer.
```

#### CTA

Primär:

```text
Kom igång
```

Sekundär:

```text
Se hur det fungerar
```

### Mobilkrav

- H1 får inte klippas i sidled.
- Texten ska vara kort.
- CTA ska synas utan lång scroll.
- Desktopmockup ska döljas, beskäras eller ersättas om den gör mobilvyn rörig.
- Hero ska inte försöka täcka alla SEO-sökord.

## 3. Problemigenkänning

### Mål

Besökaren ska känna:

> Det här är precis min vardag.

### Rubrik

```text
Känns egenkontrollen som något som alltid hamnar mellan allt annat?
```

### Innehåll

Använd korta problemkort, inte långa textstycken.

Exempel på kort:

- `Kontroller glöms bort`  
  Någon tror att någon annan redan gjort dem.

- `Pärmen är inte där den behövs`  
  Dokumentationen finns, men inte alltid samlad eller uppdaterad.

- `Ny personal vet inte vad som ska göras`  
  Rutiner sitter ofta i huvudet på den som brukar sköta dem.

- `Kontrollanten vill se historiken direkt`  
  Då ska ingen behöva leta efter papper eller rätt person.

- `Du måste fråga vem som gjort vad`  
  I stället för att bara kunna se status själv.

### SEO-värde

Det här täcker naturligt problemrelaterade sökintentioner utan keyword stuffing.

## 4. Tre huvudproblem och lösningar

### Layout

Tre tydliga block.

```text
Problem → Lösning → Resultat
```

### Block 1

#### Problem

```text
Egenkontroll tar onödigt mycket tid
```

#### Lösning

```text
Personalen ser dagens kontroller direkt och dokumenterar på plats i mobilen.
```

#### Resultat

```text
Mindre tid på dokumentation, mer tid på verksamheten.
```

### Block 2

#### Problem

```text
Det blir krångligt när kontrollanten vill se dokumentationen
```

#### Lösning

```text
Dela dokumentationen med ett knapptryck när kontrollanten kommer.
```

#### Resultat

```text
Vem som helst i personalen kan visa rätt underlag.
```

### Block 3

#### Problem

```text
Det är svårt att veta vem som gjort vad
```

#### Lösning

```text
Tidpunkt och användare loggas automatiskt med egen inloggning.
```

#### Resultat

```text
Du får överblick även när du inte är på plats.
```

## 5. Så fungerar det

### Mål

Förklara appen utan att bli teknisk.

### Steg

1. `Skapa verksamheten`  
   Välj typ av verksamhet och starta med färdiga kontroller.

2. `Personalen ser vad som ska göras`  
   Dagens kontroller visas direkt i mobilen.

3. `Dokumentera på plats`  
   Temperaturer, checklistor, foto och kommentarer sparas direkt.

4. `Visa historik vid kontroll`  
   Dela eller exportera dokumentationen när den behövs.

---

## 6. Funktioner kopplade till problem

Undvik en lösryckt funktionslista.

Använd formatet:

```text
När problemet är X → hjälper funktionen Y.
```

Exempel:

- `När kontroller glöms bort`  
  Dagens kontroller visar vad som återstår.

- `När dokumentation ligger utspridd`  
  Historiken samlas digitalt per kontroll.

- `När kontrollanten frågar`  
  Delningsvy och export gör det lätt att visa underlag.

- `När ny personal börjar`  
  Appen visar vad som ska göras utan att allt behöver förklaras muntligt.

## 7. För vilka verksamheter?

### Mål

SEO och igenkänning.

Använd separata sektioner eller kort för:

- Restaurang
- Café
- Bageri
- Foodtruck
- Livsmedelsbutik

Varje kort ska ha 2–3 meningar med verkliga exempel.

Exempel:

```text
Egenkontroll för café

Håll koll på kyltemperaturer, städning, datummärkning och dagliga rutiner utan pärmar och lösa lappar.
```

---

## 8. Kontrollant / inspektörsvy

### Mål

Visa den tydligaste stressreduceraren.

### Rubrik

```text
När kontrollanten kommer är dokumentationen redan samlad
```

### Text

```text
Du behöver inte leta efter pärmen eller vänta på rätt person. Dela dokumentationen direkt från appen och visa historik för den period som behövs.
```

---

## 9. Vanliga frågor

FAQ bör byggas kring verkliga frågor.

Exempel:

- Vad är egenkontroll för livsmedelsverksamheter?
- Måste egenkontroll dokumenteras?
- Kan egenkontroll vara digital?
- Kan personalen använda mobilen?
- Kan kontrollanten se dokumentationen?
- Behöver all personal ha egen inloggning?
- Passar det restaurang, café och bageri?

Svaren ska vara korta men hjälpsamma.

---

## 10. Kunskapsbank / fördjupning

Länka vidare till framtida eller befintliga artiklar.

Exempel:

- Egenkontroll för restaurang
- Egenkontroll för café
- Temperaturkontroll i livsmedelsverksamhet
- Digital egenkontroll
- Vad ska visas vid livsmedelskontroll?

Detta gör att landningssidan inte behöver bära all SEO själv.

---

## 11. Slut-CTA

### Rubrik

```text
Gör egenkontrollen enklare att få gjord varje dag
```

### Text

```text
Samla rutiner, dokumentation och historik på ett ställe – direkt i mobilen.
```

### CTA

```text
Kom igång
```

---

## Designriktning

- Behåll den mörka premiumkänslan om den fungerar responsivt.
- Mer luft mellan sektioner.
- Korta textblock.
- Problemkort och lösningskort ska vara skannbara.
- Mobil först.
- Desktop kan ha mockup/produktbild, men mobil ska prioriteras.

---

## Vad Codex särskilt ska undvika

- Lång hero-text.
- Text som känns skriven för sökmotorer.
- Att upprepa `digital egenkontroll` onaturligt ofta.
- Att försämra mobilvyn genom stora desktopmockups.
- Att ta bort SEO-innehåll helt utan att ersätta det längre ned.

---

## Kontrollfråga innan implementation anses klar

Kan en stressad caféägare förstå detta på mobilen inom 5–10 sekunder?

Om svaret är nej är sidan fortfarande för tung eller otydlig.
